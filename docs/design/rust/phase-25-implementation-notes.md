# Phase 25 Implementation Notes

## Summary

Phase 25 ships the TypeScript / Bun npm package `@tinyquant/core` as
a thin idiomatic camelCase wrapper over the Rust `tinyquant-core`
crate, distributed via napi-rs with the five platform `.node` files
bundled into a single fat tarball. The public API mirrors the
Phase 24 Python fat-wheel shape, so TinyQuant now ships the same
codec / corpus / backend surface across Rust, Python, and
TypeScript, versioned in lockstep. Phase 25 landed in four
sub-slices on `phase-22-25-release-chain`:

- **Phase 25.1 â€” Scaffold (LANDED at `a6c80cd`, follow-ups `d8131f0`,
  `8ff8eda`).** Added the `rust/crates/tinyquant-js/` napi-rs crate
  exporting only `version()`, the `javascript/@tinyquant/core/`
  npm package skeleton with the runtime loader, the `exports`
  map covering ESM + CJS, and the initial `build.mjs` pipeline.
  Declared three deviations from the plan (napi-rs v2 vs v3,
  Bundler module resolution vs NodeNext, `.cjs` CJS entry vs
  `.js`) â€” each is a forward-compatible substitution that the
  later slices inherit transparently.
- **Phase 25.2 â€” Codec parity + round-trip (LANDED at `7aecb5c`,
  follow-ups `ee2fade`, `51e2c96`).** Exposed `CodecConfig`,
  `Codebook`, `CompressedVector`, `RotationMatrix`, `Codec`, and
  module-level `compress` / `decompress` convenience functions
  via `#[napi]` classes. Parity fixtures (`config_hashes.json`,
  `codebook_bytes.json`, `rotation_matrix.json`,
  `round_trip_mse.json`, `brute_force_search.json`) are
  generated by `scripts/packaging/generate_js_parity_fixtures.py`
  and consumed from `tests/parity.test.ts` + `tests/round-trip.test.ts`.
  `configHash` is byte-identical to Python; round-trip MSE <
  1e-2 on 10,000 seeded random vectors.
- **Phase 25.3 â€” Corpus + backend + TS types (LANDED at `f1ccd3c`,
  follow-ups `7919e4a`, `6bad6a0`, `4aa21df`).** Exposed `Corpus`,
  `CompressionPolicy` (with `compress()` / `passthrough()` /
  `fp16()` static factories since napi-rs doesn't map Rust enums
  directly to JS consts), `VectorEntry`, `BruteForceBackend`, and
  `SearchResult`. `scripts/build.mjs` became the single entry
  point orchestrating napi build â†’ rename â†’ tsc ESM â†’
  emit-cjs-types â†’ emit-cjs-bundle; `dist/` ships six modules Ă—
  (`.js`, `.cjs`, `.d.ts`, `.d.cts`, maps) = 24+ artefacts.

  badges. Extended `COMPATIBILITY.md` to a tri-language ledger.

## Declared deviations

### From Phase 25.1 (carried forward)

#### 1. napi-rs v2 instead of v3

Plan Â§`tinyquant-js` crate pins `napi = "=3.0"` with feature
`napi9` and `napi-derive = "=3.0"`. Ship uses `napi = "2"` with
feature `napi8`. Rationale: napi-rs v3 was pre-release at
plan-authoring time; v2 is the stable line `@napi-rs/cli` 2.x
ships against. `#[napi]` attribute macros are stable across
v2/v3, so a later bump is non-breaking at the TS API layer.

#### 2. `moduleResolution: Bundler` instead of `NodeNext`

Plan Â§`tsconfig.json` specifies `"module": "NodeNext",
"moduleResolution": "NodeNext"`. Ship uses `"module": "ESNext",
"moduleResolution": "Bundler"`. Rationale: Bundler resolution
matches downstream TS + bundler toolchains (tsup, esbuild, vite)
for libraries that ship both ESM and CJS via `exports`. The
`exports` block (which drives runtime resolution) is unchanged.

#### 3. `./dist/index.cjs` instead of `./dist/index.js` for CJS

Plan Â§`package.json` names the CJS artefact `./dist/index.js`.
Ship uses `./dist/index.cjs`. Rationale: the explicit `.cjs`
extension makes Node's module-system classification unambiguous
under `"type": "module"` and avoids the consumer-set
`"type": "module"` override bug on a file named `index.js`.

### From Phase 25.3 (carried forward)

#### 4. `VectorEntry.metadata` is a no-op

Plan Â§`Corpus.insert(vector_id, vector, metadata=None)` describes
metadata as a round-trippable `Record<string, unknown>`. Ship
accepts the `metadata_json` parameter on the constructor /
`insert` / `insertBatch` for API-shape parity but the getter
unconditionally returns `null` regardless of input. Matches the
PyO3 binding's stance (`tinyquant_py/src/corpus.rs` raises
`NotImplementedError` on metadata reads/writes). No fixture or
test depends on metadata round-tripping.

### From Phase 25.4 (new)

#### 5. Build invocation in js-ci.yml

Plan Â§CI workflow (line 1153-1157) invokes `napi build --release
--target <triple> --output-dir ../../../javascript/@tinyquant/core/binaries`
directly from `rust/crates/tinyquant-js/`. Ship invokes
`npm run build` from `javascript/@tinyquant/core/` which drives
`scripts/build.mjs` (orchestrates napi + rename + tsc + emit-cjs).
Rationale: single source of truth for "how the package is built"
stays in the script; CI doesn't duplicate napi-rs CLI invocation
in YAML.

#### 6. Install-test smoke vs. full suite

Plan Â§CI workflow (line 1238) runs `node --test tests/**/*.test.ts`
inside the install-test job. Ship runs a lightweight smoke
(`require("@tinyquant/core").version()`) in install-test, and runs
the full TS suite in a separate `test-source` job on linux/x64.
Rationale: the published tarball does not contain `tests/`, so
exercising the full suite against an installed tarball requires
re-checking out the repo, which defeats "test the tarball".

#### 7. `gh` CLI vs `softprops/action-gh-release@v2` for release upload

Plan Â§Release workflow line 1329-1332 uses
`softprops/action-gh-release@v2`. Ship uses `gh release create` /
`gh release upload` via the `gh` CLI for consistency with the
0.1.1 `release.yml` hardening choice (see `CHANGELOG.md` 0.1.1
"Upgraded GitHub Actions to Node 24" entry). The CLI path is
hardened with `nullglob` against empty glob expansion and passes
the tag name via env rather than unquoted substitution.

#### 8. `publish-dry-run` companion job

Plan Â§Release workflow does not include a dry-run job. Ship adds
a `publish-dry-run` job that runs whenever `release-gate`
evaluates to false OR `inputs.dry_run == true`, so dispatch runs
and pre-release tags still exercise the full assemble â†’ pack
path and surface broken pack metadata before any real tag push.

## Forward notes

Out of scope for Phase 25, documented here so the hand-off is clean:

- **First publish (Step 12 of the plan).** Tagging `v0.1.0-rc.1`
  and publishing to the `next` dist-tag is a manual operator task.
  Prerequisites: register the `@tinyquant` scope on npm, bind
  npm OIDC trusted publishing to this repo, add the
  `@tinyquant/core` URL to the `environments.npm` config. Once
  `v0.1.0-rc.1` is pushed the `release-gate` regex
  `^v[0-9]+\.[0-9]+\.[0-9]+$` will evaluate to `false` (the
  `-rc.1` suffix fails the regex) so the first tag exercises the
  full build + pack path in `publish-dry-run` without publishing.
  The first `v0.1.0` (clean semver) tag is what actually
  publishes â€” the operator should manually verify the
  `publish-dry-run` artefact from `v0.1.0-rc.1` first.
- **`examples/typescript-consumer/`.** The plan's AC#7 requires
  a strict-TS example consumer that installs the published
  package. Not added in this branch (nothing to install until
  first publish); recommended first follow-up after the first
  `v0.1.0` publish.
- **Alpine musl target.** The loader already distinguishes
  `linux-x64-gnu` vs `linux-x64-musl` (Â§Risks glibc-vs-musl
  bullet), so adding a 6th binary is purely additive. Blocked on
  a `cross`/musl napi-build runbook; not urgent.
- **Package size gate in js-release.yml.** AC#7 (â‰¤ 10 MB gzipped,
  â‰¤ 50 MB uncompressed) is not yet enforced by a CI step. Add a
  `size-gate` step to the `publish` and `publish-dry-run` jobs
  that runs `npm pack --dry-run --json` and fails if either
  limit is exceeded.
- **Subprocess-spawn guard.** AC#12 requires a repo-root grep step
  that fails CI if the loader ever imports `node:child_process`.
  Add as a one-line step to `js-ci.yml`.
- **Tri-language `check-publish-guards` contract extension.**
  Phase 22.D / 24.3 enforce byte-identical guard expressions on
  `rust-release.yml` + `python-fatwheel.yml` via
  `cargo xtask check-publish-guards`. `js-release.yml::publish.if`
  follows the same shape but is not yet covered by the contract.
  Extend `rust/xtask/src/cmd/guard_sync_python.rs` (or add a
  sibling `guard_sync_js.rs`) to include the js-release guard so
  future edits cannot drift.
