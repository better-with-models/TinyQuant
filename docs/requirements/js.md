# JavaScript / TypeScript Binding Requirements

Functional requirements for the `@better-with-models/tinyquant-core` npm
package — the N-API binding that exposes TinyQuant to Node.js (≥ 20.10)
and Bun (≥ 1.1). Requirements in this file govern cross-language parity,
API surface stability, packaging invariants, and platform coverage.

The Python reference implementation and the Rust core are the oracles.
Where a requirement names a `Must` threshold that cannot be met by the
Python reference it is invalid and must be revised before acceptance.

> [!note] Scope
> These requirements apply to the **N-API binding** (`tinyquant-js` crate
> + `javascript/@tinyquant/core`). A future WASM package
> (`@tinyquant/core-wasm`) would have a separate requirement set.
> The Rust core is gated by [[requirements/codec|Codec Requirements]];
> this file covers only the JS-layer contract.

### FR-JS-001 — configHash is byte-identical to the Python reference

```
Gist:       The TypeScript CodecConfig.configHash value matches the Python
            reference config_hash byte-for-byte for the same (seed, dim,
            bit_width, residual_enabled) tuple.
Type:       Constraint
Actor:      Cross-language integration
Function:   For any valid combination of bit_width in {2, 4, 8},
            seed in [0, 2^64), dimension in {64, 128, 256, 512, 768, 1536},
            and residual_enabled in {true, false}, the hex string returned
            by new CodecConfig({...}).configHash shall equal the
            config_hash produced by the Python reference for the same
            parameters.
Scale:      Fraction of (seed, dim, bit_width, residual) fixture cases
            where TS configHash equals Python config_hash.
Meter:      Load the fixture file
            tests/fixtures/parity/config_hashes.json (>=120 cases);
            construct each CodecConfig in TS; assert exact string
            equality against the fixture's config_hash field.
Must:       100% of fixture cases pass.
Plan:       100%
Qualify:    seed values transmitted as BigInt to preserve u64 precision.
Rationale:  config_hash is the identity key used by corpus insertion guards
            (FR-CORP-003). A mismatch between Python and TS hashes would
            make cross-language corpus sharing impossible and would
            silently corrupt mixed deployments.
Tests:      [Node] javascript/@tinyquant/core/tests/parity.test.ts
            (120 fixture cases generated by scripts/packaging/generate_js_parity_fixtures.py)
Gap:        None.
```

### FR-JS-002 — Compress–decompress round-trip MSE within bound

```
Gist:       Compressing and decompressing N FP32 vectors through the TS
            binding produces a mean squared error below the threshold.
Type:       Performance
Actor:      Library consumer
Function:   For N FP32 vectors of dimension DIM compressed at bit_width
            BITS with residual_enabled false, the mean squared error
            between original and decompressed values over all elements
            shall not exceed the threshold below.
Scale:      MSE = (Sum (original[i][j] - decoded[i][j])^2) / (N x DIM)
            over all i in [0, N) and j in [0, DIM).
Meter:      (a) 10 000 uniform [-1, 1) vectors, dim=128, bit_width=4,
            residual=false; use mulberry32 PRNG at seed 0xcafebabe;
            train codebook on first 1 000 vectors.
            (b) 1 000 uniform [-1, 1) vectors, dim=768, bit_width=4,
            residual=false; seed 0xdeadbeef; train on first 256.
Must:       MSE < 1e-2 for both (a) and (b).
Plan:       MSE < 5e-3 for both.
Qualify:    Codebook trained on a representative sample of the same
            distribution; same CodecConfig for compress and decompress.
Rationale:  The primary correctness signal for the N-API binding is that
            the Rust codec runs end-to-end without numeric corruption
            through the JS boundary. MSE < 1e-2 at dim=128 is the Phase
            25 acceptance gate; dim=768 covers the production use case.
Tests:      [Node] javascript/@tinyquant/core/tests/round-trip.test.ts
            (covers case (a) only: N=10k, dim=128)
Gap:        GAP-JS-002 — round-trip.test.ts only tests dim=128. No test
            covers dim=768 (the production gold corpus dimension) or
            dim=1536 (the Rust perf target). A dim=768 case should be
            added as a second it() block in the same file.
            See testing-gaps.md §GAP-JS-002.
```

### FR-JS-003 — BruteForceBackend search results match Python oracle

```
Gist:       The TypeScript BruteForceBackend.search returns top-k results
            ordered and scored identically to the Python reference on the
            same fixture vectors and queries.
Type:       Constraint
Actor:      Cross-language integration
Function:   For each scenario in the parity fixture
            (tests/fixtures/parity/backend_scenarios.json), ingest the
            fixture's vectors into a TS BruteForceBackend, replay each
            query, and assert:
            (a) result length equals the Python oracle's result length.
            (b) result[i].vectorId equals oracle[i].vector_id for each i.
            (c) |result[i].score - oracle[i].score| <= 1e-6 for each i.
Scale:      (a) Fraction of queries where result length matches.
            (b) Fraction of (query, rank) pairs where vectorId matches.
            (c) Fraction of (query, rank) pairs where score is within tol.
Meter:      Load backend_scenarios.json; iterate all scenarios and
            queries; assert (a), (b), (c) for each.
Must:       100% for (a), (b), and (c).
Plan:       Same.
Qualify:    All fixture vectors are finite FP32; same dimension across
            each scenario; Python oracle was generated via the Rust-backed
            fat wheel (Phase 24 shim).
Rationale:  The brute-force backend is the default search path in
            JavaScript deployments. A ranking mismatch would produce
            wrong nearest-neighbor results silently.
Tests:      [Node] javascript/@tinyquant/core/tests/backend.test.ts
            (Python fixture-driven; covers ordering, score tolerance,
            clear(), and remove() operations)
Gap:        None.
```

### FR-JS-004 — Corpus lifecycle events match Python oracle

```
Gist:       Corpus insert, batch-insert, decompressAll, and remove
            operations emit the correct sequence of domain events and
            produce the correct vector count and containment results,
            matching the Python oracle fixtures.
Type:       Constraint
Actor:      Cross-language integration
Function:   For each scenario in tests/fixtures/parity/corpus_scenarios.json,
            construct a Corpus in TS with the scenario's CodecConfig,
            Codebook, and CompressionPolicy; perform the scenario's
            insertions (single or batch as flagged); then assert:
            (a) corpus.vectorCount equals scenario.expected_vector_count.
            (b) corpus.contains(id) is true for every id in
                scenario.expected_contains.
            (c) corpus.pendingEvents().map(e => e.type) deep-equals
                scenario.expected_events.
            (d) every event.timestamp is a Date instance.
Scale:      Fraction of (scenario, assertion) pairs that pass.
Meter:      Iterate all corpus scenarios; assert (a)-(d) for each.
Must:       100% of (scenario, assertion) pairs pass.
Plan:       Same.
Qualify:    Codebook trained on a deterministic calibration set
            (mulberry32, 512 rows) so the corpus does not need the
            Python oracle's exact training data.
Rationale:  Corpus domain events are the observable integration signal
            for downstream systems (search backends, audit logs); a JS
            consumer must see the same event sequence as a Python one.
Tests:      [Node] javascript/@tinyquant/core/tests/corpus.test.ts
            (Python fixture-driven; covers single insert, batch insert,
            decompressAll event, and remove operations)
Gap:        GAP-JS-004 — corpus.test.ts does not test:
            (1) cross-config insertion rejection (FR-CORP-003 in JS layer)
            (2) FP16 policy precision bound (FR-CORP-006 in JS layer)
            (3) compression policy immutability after first insert (FR-CORP-007)
            These policy invariants are tested in the Rust and Python layers
            but not exercised through the N-API boundary.
            See testing-gaps.md §GAP-JS-004.
```

### FR-JS-005 — Public TypeScript API surface is stable and complete

```
Gist:       All exported classes, functions, and types listed in the
            package surface are present, are the correct kind (class vs
            function vs type alias), and compile cleanly under
            TypeScript strict mode.
Type:       Constraint
Actor:      TypeScript consumer
Function:   The following symbols shall be importable from the package
            root and pass a runtime kind-check:
            Classes: CodecConfig, Codebook, CompressedVector,
            RotationMatrix, Codec, CompressionPolicy, Corpus,
            VectorEntry, BruteForceBackend, SearchResult, TinyQuantError.
            Functions: compress, decompress, version.
            Type aliases (compile-time only): ConfigHash,
            CompressionPolicyKind, StorageDtype, CorpusOpts, CorpusEvent,
            CorpusCreatedEvent, VectorsInsertedEvent,
            CorpusDecompressedEvent,
            CompressionPolicyViolationDetectedEvent.
Scale:      Fraction of listed symbols that are present, correct kind,
            and compile under `tsc --strict`.
Meter:      Compile tests/types.test.ts with `tsc -p tsconfig.test.json
            --strict --noEmit`; run under `node --test`; assert typeof
            for each class and function; assert discriminated union
            exhaustiveness for CorpusEvent.
Must:       100% of listed symbols present and type-correct.
Plan:       Same.
Qualify:    TypeScript >= 5.4 compiler; tsconfig.json as shipped in the
            package (`strict: true`, `module: NodeNext`).
Rationale:  TypeScript consumers depend on type declarations for
            compile-time safety; silent surface drift (renamed export,
            changed kind) breaks downstream builds without a runtime
            signal.
Tests:      [Node] javascript/@tinyquant/core/tests/types.test.ts
            (compile-time + runtime surface smoke; covers all 11 classes,
            3 module functions, 8 type aliases, TinyQuantError parsing,
            CompressionPolicy statics)
Gap:        None.
```

### FR-JS-006 — Runtime loader resolves correct binary for all supported platforms

```
Gist:       The runtime loader picks the correct pre-built .node binary
            for each supported (platform, arch) tuple and throws a
            descriptive error for unsupported combinations.
Type:       Function
Actor:      Library consumer on any supported operating system
Function:   On linux/x64, linux/arm64, darwin/x64, darwin/arm64, and
            win32/x64, the loader shall load the matching bundled .node
            file using require() only — no subprocess invocation. On any
            other (platform, arch) combination the loader shall throw an
            Error whose message names the unsupported combination and
            references the GitHub issue tracker.
Scale:      (a) Fraction of supported (platform, arch) pairs where the
            correct binary key is resolved.
            (b) Whether the unsupported-platform error message matches
            the required pattern.
Meter:      Unit test that monkey-patches process.platform and
            process.arch to each of the 5 supported and >=2 unsupported
            values; assert binaryKey() returns the correct string or
            throws the correct error.
Must:       (a) 100% of supported platforms resolve correctly.
            (b) Error message for unsupported platforms includes the
            platform/arch pair and a GitHub issues URL.
Plan:       Same.
Qualify:    Bundled .node binaries are present in the binaries/ directory
            (package was installed from tarball, not built from source).
Rationale:  The bundled-binaries strategy relies entirely on the loader
            for runtime platform selection; a wrong binary key causes an
            immediate segfault or module-not-found at import time.
Tests:      [Node] javascript/@tinyquant/core/tests/loader.test.ts
            (unit test monkey-patching process.platform / process.arch)
Gap:        GAP-JS-006 — The linux-arm64-musl and linux-x64-musl binaries
            are not bundled; the loader maps musl to "linux-*-musl" key but
            no corresponding .node file exists, so Alpine Linux users hit
            the missing-binary error. The loader correctly reports the
            error; the binary is the gap. A musl build should be added in
            a follow-up phase. See testing-gaps.md §GAP-JS-006.
```

### FR-JS-007 — CJS and ESM consumers can import the package

```
Gist:       The package works for both CommonJS (require) and ESM
            (import) consumers using the exports map without additional
            bundler configuration.
Type:       Constraint
Actor:      Library consumer (CJS or ESM project)
Function:   A Node.js process using require("@better-with-models/tinyquant-core")
            shall load the package and expose the same top-level symbol
            set as the ESM import path. Both entry points shall resolve
            via the `exports` map; no module.exports fallback or
            require-shim shall be needed.
Scale:      (a) CJS require succeeds and exposes CodecConfig.
            (b) ESM import resolves and exposes CodecConfig.
Meter:      (a) Run tests/cjs-smoke.test.cjs under `node --test`.
            (b) Run the main ESM test suite (round-trip, parity, etc.)
            which imports from `../dist/index.js` via ESM.
Must:       Both (a) and (b) succeed; no ERR_REQUIRE_ESM or
            ERR_UNKNOWN_FILE_EXTENSION errors.
Plan:       Same.
Qualify:    Node.js >= 20.10; package installed from the published
            tarball or built locally via `npm run build`.
Rationale:  Many Node.js server applications still use CJS (e.g. most
            Express and Fastify starters). Dropping CJS support would
            exclude a large fraction of the target audience.
Tests:      [Node] javascript/@tinyquant/core/tests/cjs-smoke.test.cjs (CJS path)
            [Node] javascript/@tinyquant/core/tests/round-trip.test.ts (ESM path)
Gap:        GAP-JS-007 — No dedicated test verifies that a .mjs consumer
            importing individual sub-paths (/codec, /corpus, /backend from
            the exports map) can resolve those entry points. Tests exercise
            the root "." export only.
            See testing-gaps.md §GAP-JS-007.
```

### FR-JS-008 — Published package size within budget

```
Gist:       The published npm tarball stays within the size budget
            defined in Phase 25 acceptance criteria.
Type:       Constraint
Actor:      npm registry, downstream CI caches
Function:   The tarball produced by `npm pack` shall satisfy:
            gzipped size <= 10 MB AND uncompressed size <= 50 MB.
            The tarball shall contain no files outside the set declared
            in the package.json `files` field.
Scale:      (a) Gzipped tarball size in MB.
            (b) Uncompressed unpacked size in MB.
Meter:      `npm pack --dry-run` during CI; parse stdout for
            "package size" and "unpacked size"; assert both thresholds.
Must:       (a) <= 10 MB gzipped.
            (b) <= 50 MB uncompressed.
Plan:       (a) <= 8 MB gzipped (current: ~4 MB).
            (b) <= 25 MB uncompressed (current: ~8.5 MB).
Qualify:    All 5 platform .node binaries are bundled; tests/ is excluded
            via .npmignore.
Rationale:  Oversized packages degrade npm install cold-start in CI and
            violate the expectations of users on metered connections. The
            50 MB hard cap is above the current actual size (~8.5 MB) but
            below the npm soft limit (~100 MB) where publish performance
            degrades.
Tests:      None.
Gap:        GAP-JS-008 — No CI step runs `npm pack --dry-run` and enforces
            the size gate. Size is documented in the Phase 25 plan but not
            automatically checked. If new binaries are added (e.g. musl,
            RISC-V) the package could exceed the budget silently. P2:
            current size is well within budget but there is no regression
            guard. See testing-gaps.md §GAP-JS-008.
```

### FR-JS-009 — Version triple-consistency enforced before publish

```
Gist:       The package version in package.json, the Rust workspace
            version in Cargo.toml, and the git release tag must all
            agree before npm publish proceeds.
Type:       Constraint
Actor:      Release workflow
Function:   The `verify-version` CI job shall compare:
            (a) javascript/@tinyquant/core/package.json `.version`
            (b) rust/Cargo.toml `[workspace.package] version`
            (c) The git tag triggering the release (stripped of the
                leading "v")
            All three shall be equal. If any differ, the job shall exit
            non-zero and block the publish step.
Scale:      Fraction of release attempts where a version mismatch is
            caught before npm publish executes.
Meter:      Inject a deliberate mismatch in a dry-run; assert the
            verify-version job exits non-zero.
Must:       100% of mismatches caught; 0 mismatched versions reach npm.
Plan:       Same.
Qualify:    Release workflow trigger is a `v*` git tag.
Rationale:  A published package whose package.json version differs from
            its Cargo.toml version makes it impossible to correlate npm
            and crates.io releases; npm provenance attestation would
            link the wrong source commit.
Tests:      [CI]  .github/workflows/js-release.yml::verify-version
Gap:        GAP-JS-009 — verify-version runs only in the release workflow;
            it does not run on PRs or pushes to main. A version mismatch
            introduced in a PR (e.g. package.json bumped but Cargo.toml
            not) would not be caught until the next release attempt.
            See testing-gaps.md §GAP-JS-009.
```

### FR-JS-010 — Loader is pure require() with no subprocess invocation

```
Gist:       The runtime loader uses only require() (or dynamic import)
            to load the .node binary; it must never invoke shell commands
            or subprocess-spawning APIs.
Type:       Constraint
Actor:      Security reviewer, sandboxed runtime environment
Function:   The source file src/_loader.ts (and its compiled dist/
            output) shall contain no reference to subprocess-spawning
            Node.js built-in APIs (the `child_process` module, or the
            shell-executing overloads of those APIs). A static grep
            of the compiled dist/ output shall find zero matches for
            these identifiers.
Scale:      Number of occurrences of forbidden identifiers in dist/.
Meter:      scripts/check_no_exec.sh greps compiled dist/ for the
            `child_process` module name and known subprocess-invocation
            API identifiers; assert exit code 1 (no matches).
Must:       0 occurrences.
Plan:       Same.
Qualify:    grep runs against the compiled dist/ output, not source,
            so minifier renaming cannot mask violations.
Rationale:  A loader that shells out could be used as an injection
            vector in sandboxed environments (AWS Lambda, Docker with
            seccomp profiles) where subprocess spawning is audited or
            blocked. The loader must be a pure require() call.
Tests:      [CI]  scripts/check_no_exec.sh (grep gate in js-ci.yml)
Gap:        GAP-JS-010 — check_no_exec.sh is documented in the Phase 25
            plan but it is unclear whether it is wired into js-ci.yml in
            the current implementation. Verify the CI job step exists and
            runs on every PR. See testing-gaps.md §GAP-JS-010.
```
