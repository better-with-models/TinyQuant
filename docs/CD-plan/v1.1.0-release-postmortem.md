# v1.1.0 Release Postmortem

> [!info] Purpose
> The v1.1.0 release was the **first end-to-end exercise** of TinyQuant's
> three-workflow publish pipeline (`release.yml`, `python-fatwheel.yml`,
> `rust-release.yml`) since those workflows were authored. Many latent
> bugs surfaced. This document captures every failure class, the root
> cause, the fix, and the pre-release check that would have caught it
> earlier — so the next major/minor release is single-digit rounds, not
> 30+ commits over 18+ hours.

> [!success] Outcome
> v1.1.0 shipped to all four registries:
>
> - **crates.io** — `tinyquant-{core,io,bruteforce,pgvector,sys,cli}` 1.1.0
> - **PyPI** — `tinyquant-rs` 1.1.0 + `tinyquant-cpu` 1.1.0
> - **npm** — `@better-with-models/tinyquant-core` 1.1.0
> - **GitHub Releases** — `v1.1.0` and `rust-v1.1.0` (with per-arch wheel
>   artifacts attached to the latter)

## Architecture recap — three workflows, four tag schemes

| Tag pattern | Workflow | Publishes to | Notes |
|---|---|---|---|
| `v*` | `release.yml` | GH Release + calibration gate | PyPI publish stages now gated off (`if: false`) — they shipped a Phase-23 placeholder wheel |
| `rust-v*` | `rust-release.yml` | crates.io (6 crates) + PyPI `tinyquant-rs` + GHCR container + GH Release | Reuses `rust-ci.yml` as the `gate` job (must be `workflow_call`-callable) |
| `py-v*` | `python-fatwheel.yml` | PyPI `tinyquant-cpu` (fat wheel assembled from `rust-v*`'s per-arch wheels) | Sequential: `rust-v*` must complete before `py-v*` is triggered |
| `@<scope>/<pkg>@<version>` *(implicit `v*`)* | `js-release.yml` | npm `@better-with-models/tinyquant-core` | Triggers off the same `v*` tag as `release.yml` |

The dependency graph: `rust-v* → py-v*`. The `py-v*` tag must be pushed
**after** `rust-release.yml` completes successfully and uploads
`tinyquant_rs-*.whl` artifacts to the `rust-v*` GitHub Release.

## Failure classes (chronological)

### F1. Latent: `rust-ci.yml` not exposed as `workflow_call`

**Symptom.** `rust-release.yml`'s `gate` job invokes
`uses: ./.github/workflows/rust-ci.yml` to re-run every per-PR check on
the tagged commit. The reusable-workflow contract requires the called
workflow to declare `on: workflow_call:`. `rust-ci.yml` only had
`pull_request` and `push`. GitHub reported "This run likely failed
because of a workflow file issue" with **zero check runs**.

**Fix.** Add `workflow_call:` to `rust-ci.yml`'s `on:` block (commit
`672295a`).

**Pre-release check.** Add an xtask or pre-commit gate that resolves
every `uses: ./.github/workflows/X` reference and asserts `X` declares
`workflow_call:`.

### F2. Toolchain pin drift: dev tools require post-MSRV editions

**Symptom.** Multiple jobs failed with:

```text
feature `edition2024` is required
The package requires the Cargo feature called `edition2024`, but
that feature is not stabilized in this version of Cargo (1.81.0)
```

Affected: `cargo install cargo-vet`, `cargo install cargo-audit`,
`cargo install cargo-auditable`, `cargo install cross`, `cargo publish
-p tinyquant-sys` (the `Verifying` step).

**Root cause.** Workspace MSRV is **1.81** (a deliberate gate for
runtime consumers), but dev/auditor tooling has moved on — `cargo-vet`
0.10+ needs 1.82, `cargo-audit` 0.22+ needs 1.85, latest
`cargo-auditable`/`cross` need 1.85, and tinyquant-sys's bindgen
chain transitively pulls a crate that needs `edition2024` (1.85+) at
verify-build time.

**Fix.** Bump `RUSTUP_TOOLCHAIN` to `1.87.0` at the **job level** (not
workspace level) for jobs that only run dev/auditor tooling
(`supply-chain-gate`, `publish-crates`) or where an installer requires
edition2024 (`cargo-auditable` / `cross` install steps in `build`
matrix). Runtime MSRV stays 1.81 — only the build/publish toolchain
moves. Commits `51f3dbe`, `98671f2`, `4534f36`.

**Pre-release check.** Add a CI job that runs `cargo install
cargo-{vet,audit,auditable} cross --locked --dry-run` under the
declared MSRV and fails loudly when edition drift breaks installation.
Update the MSRV proactively, or pin the auditor tools to a specific
older version.

### F3. Pre-existing security vulnerability in transitive dep

**Symptom.** `cargo audit` failed with:

```text
RUSTSEC-2025-0020 — pyo3 < 0.24.1: Risk of buffer overflow in
PyString::from_object
```

**Root cause.** We pin `pyo3 = 0.22`. The advisory's exploitable surface
is `PyString::from_object` which we do not call directly. Upstream 0.22
is end-of-life; the fix requires the 0.22 → 0.24 `Bound`-API migration
(non-trivial code change).

**Fix.** Document in `rust/audit.toml` and pass `--ignore RUSTSEC-2025-0020`
to `cargo audit` (commits `aa34b06`, `34ed1e1`). Drop `--deny warnings`
so unmaintained-crate notices (`paste`, `number_prefix`) don't fail the
gate either.

**Open follow-up.** Schedule pyo3 0.22 → 0.24 migration for v1.2.0 and
remove the ignore.

**Pre-release check.** Run `cargo audit` weekly on `develop` so new
vulns surface between releases, not at tag-push time.

### F4. R19 fixture flakiness re-emerged across runner pool

**Symptom.** `seed_42_dim_768_matches_frozen_snapshot_bit_for_bit` failed
on every Linux x86_64 / aarch64 / macOS / Windows host with ~529 K of
~590 K f64 words diverging. The fixture had been green on `develop`
between 2026-04-21 and 2026-04-23 with the AVX2 ISA cap from `e04ce5c`.

**Root cause.** GitHub-hosted runner pool started producing
bit-different `dim=768` QR output even with AVX2 dispatch pinned — the
cap reduced but did not eliminate cross-runner variation in faer 0.19
parallel Householder reduction. Independent of any TinyQuant commit.

**Fix.** Re-`#[ignore]` the bit-exact assertion (commit `2701fb4`); rely
on the orthogonality companion test for semantic correctness. R19
remains open: the proper fix is faer 0.20+'s per-call `Parallelism` API
plus a fixture regenerated under a known scalar QR.

**Pre-release check.** Track R19 in
[[design/rust/risks-and-mitigations]] and treat any *resurrection* of
the bit-exact assertion as a separate engineering task with its own
fixture-stability validation.

### F5. Self-hosted runner referenced but not registered

**Symptom.** `release.yml`'s 6 jobs all targeted
`runs-on: docker-runner` and queued for 4+ hours with `runner: NONE`.
`gh api .../actions/runners` confirmed **zero self-hosted runners
registered** against the repo.

**Root cause.** The self-hosted runner that was the documented release
host had been decommissioned; the workflow was never updated.

**Fix.** Switch all 6 jobs to `runs-on: ubuntu-22.04` (commit `b7a9303`).
Documented in-line that PyPI publishing already uses OIDC trusted
publishing (no token leak risk on GitHub-hosted), and the `pypi` /
`testpypi` environments carry their own approval gate.

**Pre-release check.** A pre-tag hook should run `gh api
.../actions/runners` and fail if any workflow's `runs-on:` references
a self-hosted label that has zero registered runners.

### F6. Verify-tag step parsed dynamic Python `__version__` literally

**Symptom.** `release.yml`'s `Verify Tag` job failed with the message
"Tag v1.1.0 != __init__" followed by an empty string. The shell sed
regex couldn't extract a
version from:

```python
__version__: str = _core.__version__
```

**Root cause.** `src/tinyquant_cpu/__init__.py` and the production
fat-wheel shim defer `__version__` to the compiled Rust extension at
runtime — there is no static literal to grep. The sed regex
`^__version__ = "..."` matched zero lines, so the shell variable became
empty and the equality check failed.

**Fix.** Drop the `__init__.py` version check; rely on `pyproject.toml`
(the source of truth maturin flows into `_core.__version__`). The
`scripts/check_version_consistency.sh` PR-time check covers
Cargo-workspace ↔ npm-package.json drift separately. Commits `2201c5a`,
`2ca811c`.

**Pre-release check.** When updating `__version__` to a dynamic source,
audit every file that *parses* it textually. A `grep -rn '__version__'`
across `.github/workflows/`, `scripts/`, and `Makefile`-equivalents is
trivial.

### F7. `release.yml` published an empty Phase-23 placeholder wheel

**Symptom.** `release.yml`'s `Publish to TestPyPI` step succeeded
uploading a 10 KB wheel, then `Verify installation from TestPyPI`
failed with `ModuleNotFoundError: No module named 'tinyquant_cpu'`.

**Root cause.** The repo-root `pyproject.toml` is the **dead-wheel
guard** from Phase 23 (CI's `Wheel leakage guard` job enforces it).
Building it produces a deliberate zero-content placeholder wheel.
`release.yml` was authored before Phase 24 introduced
`python-fatwheel.yml` as the real PyPI publisher and was never updated
to stop publishing the placeholder.

**Fix.** Apply `if: false` to `release.yml`'s `publish-testpypi` and
`publish-pypi` jobs with an in-line comment pointing to
`python-fatwheel.yml` (commit `6213fa3`). The other jobs (verify-tag,
calibration, build, GH-release) remain useful as a cross-language
release gate.

**Pre-release check.** A workflow-architecture doc enumerating
*which workflow publishes which artifact for which tag* would have
caught this. (See "Pre-release checklist" below.)

### F8. macos-13 (Intel) runner pool throttled

**Symptom.** Every `runs-on: macos-13` job — `rust-release.yml`'s
`Build (x86_64-apple-darwin)`, `js-release.yml`'s `build .node
(x86_64-apple-darwin)`, `python-fatwheel.yml`'s `install-test
(macos-13, macos_x86_64)` — queued >25 min with `runner: NONE`. macos-14
(Apple Silicon) jobs allocated immediately.

**Root cause.** GitHub-hosted macos-13 (Intel) runner availability
collapsed in late 2026.

**Fix.**

- For `rust-release.yml` Build matrix: switch x86_64-darwin to
  `os: macos-14` and let cargo cross-compile via `--target
  x86_64-apple-darwin` (Apple supports darwin→darwin natively).
- For `js-release.yml`: same switch + add `NAPI_CROSS_TARGET` env so
  napi-rs cross-compiles correctly (mirrors `js-ci.yml`).
- For `python-fatwheel.yml`'s install-test: drop the macos_x86_64
  matrix entry entirely. Rosetta isn't available on macos-14 GitHub
  runners and Python is arm64-native, so cross-arch import-test is not
  possible. The wheel is still built/signed/uploaded by
  `rust-release.yml`'s Build matrix.
- Commits `61bc56e`, `aca119b`.

**Pre-release check.** Periodically (monthly?) check the GitHub
runner-images deprecation calendar and migrate matrix entries before
they go fully throttled.

### F9. i686 Windows wheel needs a Python interpreter that isn't installed

**Symptom.** `rust-release.yml`'s `Build (i686-pc-windows-msvc)` failed at
the `Build tinyquant-py wheel` step:

```text
Caused by: Need a Python interpreter to compile for Windows without
PyO3's `generate-import-lib` feature
```

**Root cause.** GitHub-hosted Windows runners ship 64-bit Python only;
no 32-bit Python interpreter is on PATH. PyO3's compile path needs one
unless `generate-import-lib` is enabled.

**Fix.** Skip the wheel build/upload/attest steps for
`i686-pc-windows-msvc` (commits `6f2bdf0`, `cec052b`). The CLI and sys
crates still build for that target. We don't ship 32-bit Windows
wheels — PyPI's `tinyquant-cpu` only publishes 64-bit Windows.

**Pre-release check.** Document the wheel-target matrix explicitly:
which OS×ABI combos are shipped and which are deliberately skipped.

### F10. Crates.io Trusted Publisher first-publish bootstrap

**Symptom.** `cargo publish` returned `invalid-publisher` Trusted
Publisher claim mismatch on the first attempt.

**Root cause.** Crates.io Trusted Publishers must be configured per
crate before the first publish. For never-before-published crates, the
"Pending Publishers" feature lets you pre-register the claim; for
already-published crates, the claim is added at the per-crate Settings
page. The configuration must match exactly: workflow filename,
environment name, repository owner, repository name.

**Fix.** Configured Trusted Publishers on crates.io for all 6 crates
with workflow=`rust-release.yml`, environment=`cargo`. The workflow
already targeted `environment: cargo` (see line ~503).

**Pre-release check.** A pre-tag script (or release-checklist item)
should verify each PyPI/crates.io project has the expected Trusted
Publisher claim *before* tag push. The check can hit the public APIs
without auth for the basic existence test.

### F11. Wheels uploaded as workflow artifacts but not attached to GH Release

**Symptom.** `python-fatwheel.yml`'s `fetch-inputs` failed:

```text
no assets match the file pattern
```

after `gh release download "rust-v1.1.0" --pattern 'tinyquant_rs-*.whl'`.
The wheels existed as `wheels-*` workflow artifacts but were never
attached to the `rust-v1.1.0` GitHub Release.

**Root cause.** `rust-release.yml`'s `publish-release` step only
attached CLI archives + libs + supply-chain report to the GH Release.
Wheels were uploaded via `actions/upload-artifact` (workflow-scoped,
14-day retention, doesn't cross workflow runs) but not promoted to the
Release.

**Fix.** Add `wheels-*` artifact download + `wheels/*.whl` to the
`softprops/action-gh-release` files glob (commit `cd7c524`). For v1.1.0
itself, the wheels were attached manually via `gh release upload`.

**Pre-release check.** Test the `python-fatwheel.yml` ↔ `rust-release.yml`
contract end-to-end on a `dry_run` cycle before a real release: push a
disposable `rust-vX.Y.Z-rc1` tag, verify the GH Release has all
expected `tinyquant_rs-*.whl` files, then run `python-fatwheel.yml`
with `workflow_dispatch` and `dry_run=true`.

### F12. Fat-wheel assembler regex couldn't parse compound platform tags

**Symptom.** `assemble` step reported:

```text
missing required platform wheels: ['linux_aarch64_gnu', 'linux_x86_64_gnu']
```

even though both wheels were present in `dist/input/`.

**Root cause.** `WHEEL_NAME_RE`'s platform group used `[^.]+`, which
matches up to the first `.`. Modern maturin emits compound platform
tags joined with `.` per PEP 425:

```text
tinyquant_rs-1.1.0-cp312-abi3-manylinux_2_17_x86_64.manylinux2014_x86_64.whl
```

The regex failed to match the whole filename and the wheel was
silently `continue`d in the parsing loop.

**Fix.** Switch to `[^-]+?` (lazy any-not-dash) and split the captured
compound on the first `.` for `PLATFORM_KEY_BY_TAG` lookup. Also
expand the dict to cover the actual tags maturin emits today
(`macosx_10_12_x86_64`, `manylinux_2_17_aarch64`,
`musllinux_1_2_aarch64`). Commits `701b054`, `3a91af4`.

**Pre-release check.** Add a unit test that exercises
`WHEEL_NAME_RE` against the real wheel-filename outputs of every
build-matrix entry. The packaging test suite under `tests/packaging/`
already had this scaffolding — extend its corpus to include compound
tags.

### F13. Hard-coded `Version: 0.2.0` in METADATA template

**Symptom.** PyPI returned:

```text
400 Version in filename should be '0.2.0' not '1.1.0'.
```

**Root cause.** `scripts/packaging/templates/METADATA` carried
`Version: 0.2.0` from when the template was authored at Phase 24. The
assembler embedded the template raw into the wheel's PKG-INFO without
substituting the actual release version. Filename version (`1.1.0`)
disagreed with internal METADATA version (`0.2.0`); PyPI's
`_check_filename` validator rejected.

**Fix.** Substitute the `Version:` line in `metadata_src` with the
`--version` argument before embedding (commit `5c00f8b`). The literal
`0.2.0` in the template now serves as a placeholder; runtime
substitution is the single source of version truth.

**Pre-release check.** Twine has a `check` step that catches some
metadata issues. Run `twine check` against the assembled fat wheel
**before** the publish step so this kind of mismatch is caught at
build time, not by PyPI's response. Already partially in place; needs
to be tightened to fail on version mismatches.

### F14. Structurally-broken binary-count gate

**Symptom.** `Binary-count gate (exactly N)` failed silently. The shell
pipeline `python -m zipfile -l <whl> | grep -cE '_core\.(abi3\.so|pyd)$'`
always returned 0.

**Root cause.** `python -m zipfile -l` outputs `<name>  <date>  <size>`
per row. Anchoring the regex on `$` matched the size column, never the
binary name. The gate had been broken since it was written; previous
runs aborted earlier in the assemble step so the bug never surfaced.

**Fix.** Switch to `python -c` reading `zipfile.ZipFile.namelist()`
directly — same regex, applied to actual path strings. Commit
`6f2c551`. Also bumped the count from 5 → 6 to include the
`musllinux_1_2_x86_64` lane that was always supposed to ship.

**Pre-release check.** `dry_run` rehearsal would catch this. More
generally, any "exactly-N" gate must be self-consistent with the
matrix that produces N — so the matrix should either feed N into the
gate as a derived value, or the gate's expected N should live next to
the matrix definition.

### F15. Windows pip self-upgrade requires `python -m pip`

**Symptom.** `install-test (windows-2022, win_amd64)` failed at
`pip install --upgrade pip` with:

```text
ERROR: To modify pip, please run the following command:
<python>.exe -m pip install --upgrade pip
```

**Root cause.** pip's Windows wrapper (`.exe`) cannot replace itself
in-place on the same filesystem.

**Fix.** Always invoke `python -m pip` for the self-upgrade — works on
both POSIX and Windows. Commit `dfc8819`.

**Pre-release check.** Standard idiom; codify in a project pip-policy
note so all workflows use it consistently.

### F16. npm OIDC trusted-publisher auth requires npm CLI 11.5.1+

**Symptom.** `js-release.yml` Publish job repeatedly failed with
`404 Not Found - PUT https://registry.npmjs.org/tinyquant` even though:
the trusted publisher was correctly configured on npmjs.com
(repo + workflow filename + environment all matching the OIDC claims),
the workflow had `id-token: write` permission at job level, and the
provenance attestation was successfully signed and published to the
Sigstore transparency log earlier in the same step.

**Root cause.** Two compounding bugs:

1. **`actions/setup-node@v4` auto-injects `NODE_AUTH_TOKEN`** when
   `registry-url` is set. With no `NPM_TOKEN` secret defined, it
   defaults to `${{ github.token }}` — a `ghs_…` token that npmjs.com
   doesn't recognise — masquerading as the npm auth token via
   `_authToken=${NODE_AUTH_TOKEN}` in `~/.npmrc`. Result: npm sends a
   GitHub token to npm, gets 404.

2. **npm CLI 10.x (the default that ships with Node 20.x) supports
   provenance *signing* but not OIDC *auth*.** Trusted-publisher
   token exchange landed in npm **11.5.1**. Even after `unset
   NODE_AUTH_TOKEN` cleared the GitHub-token contamination, npm 10.x
   could not exchange the OIDC token for a publish credential and
   returned 404. The Sigstore signing succeeded because that path
   only requires npm 9.5+.

**Fix.** Combined: (a) `unset NODE_AUTH_TOKEN` immediately before
`npm publish` to discard the `setup-node`-injected GitHub token, and
(b) `npm install -g npm@latest` in the same shell to upgrade past
11.5.1 before publishing.

**Pre-release check.** Treat npm trusted publishing as having a hard
floor of npm **11.5.1**. Any workflow that uses `npm publish
--provenance --access public` against a trusted-publisher-configured
package must explicitly upgrade npm in the publish step — do not rely
on the bundled npm in `actions/setup-node`. Add a CI smoke that asserts
`$(npm --version | head -c2) -ge 11` before any provenance publish.

## Pre-release checklist (apply for every tag push)

Before pushing any release tag (`v*`, `rust-v*`, `py-v*`):

- [ ] **Toolchain-pin sweep.** For every `cargo install <tool>` in the
      release workflows, dry-run the install under MSRV and verify it
      succeeds. Bump per-tool toolchain overrides if anything fails on
      `edition2024`.
- [ ] **npm CLI floor (11.5.1+).** For any workflow that publishes to
      npm via OIDC trusted publishing, confirm the publish step runs
      `npm install -g npm@latest` (or pins npm ≥ 11.5.1) before
      `npm publish --provenance`. The npm bundled with Node 20 is
      10.x and silently fails OIDC auth with 404.
- [ ] **Reusable-workflow contract.** For every `uses: ./.github/workflows/X`,
      `grep -q 'workflow_call:' X` to confirm it's callable.
- [ ] **Self-hosted runner availability.** `gh api
      /repos/$OWNER/$REPO/actions/runners --jq '.total_count'` should
      be non-zero if any workflow targets a self-hosted label, and
      should match `gh api .../actions/runs/$RUN_ID/jobs --jq
      '[.jobs[] | select(.runner_name == null and .status == "queued")] | length'` returning 0 within ~5 minutes of the
      tag push.
- [ ] **Trusted-publisher claim alignment.** For every PyPI / crates.io
      project to be published, verify the registered Trusted Publisher
      claim matches the actual workflow filename and environment name.
      (PyPI's API: `GET https://pypi.org/manage/project/<NAME>/` —
      requires session auth; for read-only existence check
      `GET https://pypi.org/pypi/<NAME>/json`. crates.io similarly.)
- [ ] **`cargo audit` is green** (or every ignored advisory is in
      `rust/audit.toml` with documented rationale + planned removal
      release).
- [ ] **`twine check`** the fat wheel locally to catch METADATA /
      filename / Description-Content-Type mismatches before PyPI gets
      a chance to reject.
- [ ] **Pending Publisher version pin (PyPI).** If any project is using
      a *Pending* Publisher (project not yet on PyPI), confirm the
      pinned version field equals the version about to be released.
      The first publish consumes the pending entry — afterwards normal
      Trusted Publisher rules apply.
- [ ] **Sequencing for `py-v*`.** Push `rust-v*` first, **wait for
      `rust-release.yml` to complete successfully and create the GH
      Release with `tinyquant_rs-*.whl` artifacts attached**, then push
      `py-v*`. (Future improvement: have `python-fatwheel.yml` poll
      `rust-release.yml`'s outputs instead of a raw `gh release
      download`, with explicit retry-with-backoff.)
- [ ] **Dry-run rehearsal.** For non-trivial workflow edits, push a
      disposable `*-rc1` tag (or `workflow_dispatch` with
      `dry_run=true` where supported) before the real tag.

## Architectural improvements queued

- [ ] **Workflow contract test** (`xtask check-publish-guards` already
      covers `rust-release.yml` ↔ `python-fatwheel.yml` `if:`
      expressions; extend it to also assert
      `workflow_call:` exists on every reusable workflow).
- [ ] **Promote wheels to GH Release artifacts** as part of
      `rust-release.yml`'s `publish-release` step. ✓ (Commit
      `cd7c524` — already shipped for v1.1.0+.)
- [ ] **METADATA template uses a sentinel** (e.g., `Version: 0.0.0+TBD`)
      and the assembler errors at build time if the substitution
      didn't happen, rather than silently shipping a stale literal.
- [ ] **Binary-count gate parameterised** by the matrix that produces
      the inputs. Single source of truth for N.
- [ ] **Replace ad-hoc "exactly N" assertions** with explicit per-key
      presence/absence assertions ("contains `_lib/win_amd64/_core.pyd`"
      etc.), so adding/removing a target is one matrix edit + one
      assertion edit, not two arithmetic-coupled places.
- [ ] **Pyo3 0.22 → 0.24 migration** (tracked separately, scheduled for
      v1.2.0; closes RUSTSEC-2025-0020 ignore in `rust/audit.toml`).
